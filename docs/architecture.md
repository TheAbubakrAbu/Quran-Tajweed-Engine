# Architecture

How the engine is put together, and why. Read this once and the whole repo makes sense.

## Design principles

1. **Data-first.** The canonical product is the JSON in [`/data`](../data). Code is replaceable; the data
   is the asset. Any language reads JSON, so the engine is automatically cross-platform.
2. **Pure & offline.** No engine function needs a network, database, or framework. Audio is the only thing
   that touches the network, and even that is just URL strings you hand to your own player.
3. **One contract, many ports.** Every language implementation follows the same
   [porting contract](PORTING.md), so the API feels familiar everywhere.
4. **Specs over magic.** Each feature has a written spec (`docs/01…08`) precise enough to reimplement from
   scratch. The reference code is the executable version of those specs.
5. **Respect the text.** The Quranic Arabic is sacred and shipped unmodified. Translations and tafsir keep
   their attribution.

## Layers

```
        ┌──────────────────────────────────────────────────────────────┐
        │  YOUR APP  (SwiftUI / Compose / React / Flutter / CLI / …)    │
        └──────────────────────────────┬───────────────────────────────┘
                                        │  imports a port (or reads data)
        ┌──────────────────────────────▼───────────────────────────────┐
        │  PORT  (packages/quran-engine-*)                              │
        │  Quran · Tajweed · JuzPage · Audio · Search · Sorting · Cache │
        │  thin, idiomatic wrappers + a few pure functions             │
        └──────────────────────────────┬───────────────────────────────┘
                                        │  loads
        ┌──────────────────────────────▼───────────────────────────────┐
        │  DATA  (/data)                                                │
        │  quran.json · surah-info.json · names-of-allah.json           │
        │  juz.json · reciters.json · tajweed-rules.json                │
        │  qiraat/*.json · surahs/NNN.json · tajweed/NNN.json           │
        └──────────────────────────────────────────────────────────────┘
```

## The data files

| File | Role | Derived? |
|---|---|---|
| `quran.json` | Source of truth: all text, translations, per-ayah juz/page/counts | original |
| `surah-info.json` | "About this surah" essays | original |
| `names-of-allah.json` | 99 Names | original |
| `qiraat/qiraah-*.json` | 7 alternate readings | original |
| `juz.json` | 30 juz boundary names + ranges | derived from Swift `juzList` |
| `reciters.json` | 62 reciters (audio feeds) | derived from Swift reciter tables |
| `tajweed-rules.json` | Rule catalogue: categories, colors, trigger letters | derived from `TajweedRules.swift` |
| `tajweed-annotations.json` + `tajweed/NNN.json` | Pre-computed tajweed spans for all 6236 ayahs | generated by the JS detector |
| `surahs/NNN.json` + `surahs/index.json` | Per-surah split for lazy/web loading | generated from `quran.json` |

Derived/generated files are reproducible: `node scripts/build-data.mjs` regenerates the per-surah split and
the tajweed corpus; the derivation of `juz.json` / `reciters.json` / `tajweed-rules.json` is documented in
[CREDITS.md](../CREDITS.md) and [data/README.md](../data/README.md).

## Two ways to do tajweed

This is the one place with real algorithmic depth, so the engine offers two strategies:

- **(A) Consume the corpus** — load `tajweed-annotations.json`, map each `rule` to a color. Tiny, exact,
  consistent. **All native ports use this.** Best when you're coloring the bundled Hafs text.
- **(B) Run the detector** — the JS package ([`src/tajweed.js`](../packages/quran-engine-js/src/tajweed.js))
  implements the full heuristic engine from [docs/02](02-tajweed.md). Use it to color text the corpus
  doesn't cover (other qiraat, user input) or to avoid a data dependency.

The corpus is literally the detector's output over all 6236 ayahs, so (A) and (B) agree by construction.

## Module map (per port)

Every port exposes the same surface:

| Module | Responsibility | Spec |
|---|---|---|
| `quran` | surahs, ayahs, translations, qiraat text, global ayah number | 01 |
| `tajweed` | colored spans (via corpus or detector) | 02 |
| `juzPage` | juz & page navigation | 03 |
| `audio` | reciter directory + surah/ayah URL builders | 04, 05 |
| `search` | verse & surah search, reference parsing | 06 |
| `sorting` | sort & filter the 114 | 07 |
| `cache` | offline-download paths | 08 |
| `text` | Arabic normalization, grapheme clustering, UTF-16 slicing | — |
| `engine` | facade tying it together + the disk loader | — |

## String offsets: the one cross-language gotcha

Tajweed annotation `start`/`end` are **UTF-16 code-unit offsets** (the native unit of JS, Swift, Java,
Kotlin, Dart, C#). Languages that index by code point (Python) or bytes (Rust, Go) must convert — every
such port ships a `utf16_slice` helper, and the porting contract spells it out. See
[PORTING.md → String indexing](PORTING.md#string-indexing-utf-16-offsets).

## Performance notes

- Loading `quran.json` (~5 MB) is the heaviest step. For web, prefer the per-surah `surahs/NNN.json` files
  and load on demand; the lightweight `surahs/index.json` (no ayah text) is enough to render a surah list.
- Search builds an in-memory index of 6236 short blobs at construction. That's fast enough to filter
  linearly; the reference Swift app additionally keeps inverted token/prefix indexes for very hot search.
- Tajweed via the corpus is a dictionary lookup + a slice — effectively free.

## Extending the engine

- **New language port?** Follow [PORTING.md](PORTING.md). Implement Load + Quran + Tajweed(corpus) +
  JuzPage + audio URLs + sorting + reference parsing, ship a test asserting the canonical cases, done.
- **New data (translations, tafsir, word-by-word)?** Add a JSON file under `/data`, document its schema in
  a new `docs/` page, and surface it in the ports. Keep the Arabic text untouched.
- **Better tajweed?** Improve `src/tajweed.js`, regenerate the corpus with `scripts/build-data.mjs`, and
  every port benefits for free.
