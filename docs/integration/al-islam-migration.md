# Migrating Al-Islam onto the engine

Goal: **Al-Islam stops maintaining its own Quran data + algorithms and consumes `quran-engine-swift` as a package.** This is the roadmap. It is staged so each step ships independently and nothing regresses — the in-app swap is the *last* thing, after the engine has parity and is protected.

> Status: the engine side is being *prepared*. The actual edits to the Al-Islam app are deferred until you give the word.

## Principle: what the engine owns vs. what the app keeps

| Engine owns (pure data + algorithms) | App keeps (UI + user state) |
|---|---|
| Quran text (Hafs + qiraat), translations, transliteration | SwiftUI views, navigation, animations |
| Juz/page navigation, `juzStats`, `juzFromEnd` | Bookmarks, favorites, notes |
| Search (verse + surah + reference, boolean grammar) | Last-read / last-listened / khatm / history |
| Tajweed **per-character rule + color** (precomputed for all 6236 ayahs) | Tajweed **rendering**: `AttributedString` build, beginner spacing, diacritic-strip projection, per-category visibility toggles |
| Reciters + audio URLs, sorting, Arabic text utilities | Settings/preferences, fonts, color theme |

The tajweed split is the important one: the engine already ships **every rule for every ayah** as precomputed spans (`tajweed-annotations.json`, generated by the full JS detector — madd, tafkhim, ghunnah, qalqalah, iqlaab, silent, muqattaat, the lot). The app keeps only the *presentation* layer. So Al-Islam's ~1000-line tajweed scanner in `QuranData.swift` is replaced by "ask the engine for spans, color them."

## Packaging — already done

`quran-engine-swift` is now self-contained: the JSON corpus is bundled as a SwiftPM resource (`Sources/QuranEngine/Resources/`, generated from `/data` by `scripts/sync-package-resources.mjs`). `try Engine.load()` works with **no filesystem setup** — Al-Islam adds the package and calls it. See [ios.md](ios.md).

## Phased plan

**Phase 0 — protect (done).** Shared conformance vectors (`/conformance/vectors.json`) run in all 7 ports; CI (`.github/workflows/ci.yml`) runs every port's tests and verifies the generated artifacts (annotations, bundled resources) stay in sync with `/data`. This is the safety net the swap relies on.

**Phase 1 — close the parity gaps in the engine** (the engine must expose everything Al-Islam reads). Prioritized below.

**Phase 2 — adopt drop-in equivalents in Al-Islam** (the already-100%-compatible areas): text access, juz/page, reciters/audio, sorting, Arabic-text utilities. Route `QuranData` through `engine.*` while keeping the same `QuranData` public surface, so views don't change yet.

**Phase 3 — swap tajweed.** Replace the in-app detector with `engine.tajweed.tajweedSpans(surah, ayah)`; keep the `AttributedString` builder, projection, beginner spacing, and visibility toggles in the app, driven by engine spans. Gate with a side-by-side visual diff (the conformance `tajweed` vectors + a screenshot pass over a sample of ayahs).

**Phase 4 — delete** the now-dead Quran data/algorithm code from Al-Islam; `QuranData` becomes a thin `@MainActor` store wrapping the engine + app state.

## API mapping (Al-Islam `QuranData` → engine)

| Al-Islam | Engine |
|---|---|
| `quranData.surah(id)` / `ayah(surah:ayah:)` | `engine.quran.surah(id)` / `engine.quran.ayah(_,_)` |
| `ayah.textArabic(for: qiraah)` / `textCleanArabic` | `engine.quran.arabicText(_,_, riwayah:)` / `cleanArabicText(...)` |
| `juzStats(for:)`, `ayahs(inJuz:)`, `surahs(inJuz:)`, `firstAyahResult(page:juz:)` | `engine.juzPage.juzStats / ayahsInJuz / surahsInJuz / firstAyahOfPage|Juz` |
| `searchVerses(term:limit:offset:)` | `engine.search.searchVerses(_, offset:limit:ignoreSilentLetters:)` |
| `filteredSurahs(query:)`, `resolveSurahIdentifier`, `parse "2:255"` | `engine.search.searchSurahs(_)` / `parseReference(_)` |
| reciter list + audio URLs | `engine.reciters.*`, `surahAudioURL`, `ayahAudioURL` |
| sort modes, makkan/madani filter | `sortSurahs(_,_,_)`, `filterByRevelationType(_,_)` |
| Arabic normalization helpers | `ArabicText.*` |
| tajweed `AttributedString` | `engine.tajweed.tajweedSpans(_,_)` → app renders |

## Parity gaps to add to the engine first (Phase 1)

From the audit. Pure data/algorithm items the engine should own before the swap. Each should land in the JS reference + the other ports + a conformance vector (so it can't drift):

**Low effort / high value — ✅ DONE (batch 1, all 7 ports + conformance vectors)**
- ~~**Sajdah ayahs**~~ — `Quran.sajdahAyahs()` / `isSajdahAyah(s,a)`; the ۩ mark is **U+06E9** (the audit's U+06D9 was wrong) — exactly 15 ayahs.
- ~~**Surah info accessor**~~ — `Quran.info(surahId)` now in every port; `surah-info.json` loads by default.
- ~~**Names of Allah accessor**~~ — `engine.namesOfAllah` (`all()`, `byNumber(n)`); `names-of-allah.json` loads by default.
- ~~**Count-based surah filter**~~ — `filterByCounts(surahs, { ayahs, pages })` with `{op,value}` predicates.
- **From-the-end shorthand** — `Quran.surahFromEnd(n)` added (joins `JuzPage.juzFromEnd`); page-from-end is `firstAyahOfPage(totalPages()+1-n)`. The full search-bar `-N` *routing* (which keyword → page/juz/surah) stays app-side — the engine ships the primitives.

**Medium — ✅ mostly DONE (batch 2)**
- ~~**Muqattaat**~~ — `engine.muqattaat` (`all()`, `pronunciation(s,a)`, `letterName(c)`) over the new `data/muqattaat.json`; vocalized names keep the madd-lāzim maddah so tajweed colours them.
- ~~**Surah computed flags**~~ — `Quran.pageChangesWithinSurah / juzChangesWithinSurah / pageOrJuzChangesWithinSurah`.
- ~~**Reciter display-name variants**~~ — already covered by `ayahNowPlayingName(reciter)` (= `name (qiraah)` or `name`), which is exactly Al-Islam's `displayNameWithEnglishQiraah` / `displayNameForNowPlaying`.
- ~~**Qiraah existence / per-qiraah ayah counts**~~ — ✅ DONE. `Quran.existsInQiraah(surah, ayah, riwayah)` + `numberOfAyahsInQiraah(surah, riwayah)`. Turned out to be a simple accessor after all: Al-Islam's qiraah "merge" (`loadQiraatOverlay`) is **pure id-matching** — a Hafs ayah exists in a riwayah iff that riwayah's feed has an entry with the same id — and the feeds are numbered contiguously `1..count`, so a per-surah **count** fully determines existence. Shipped as the compact `data/qiraat-counts.json` (~6.5 KB, generated by `build-data.mjs`), loaded by default (no 11 MB qiraah text needed). *(Note: the engine's `arabicText(_,_,riwayah)` still id-matches too — same behaviour as the app, which is what we want for parity.)*

**App-only (do NOT move to the engine)**
- Bookmarks, favorites, notes, last-read/listened, khatm, history, search history.
- Tajweed *rendering*: `AttributedString`, beginner spacing, diacritic-strip projection, per-category visibility, caching.
- Fatiha ta'awwudh/Bismillah special *display* (`displayArabicText`), boundary-divider view models (`boundaryModel`), display labels (`ayahCountLabel`), font/theme.
- **Search-box routing** — which typed keyword maps to page/juz/surah/count, and juz-name resolution (`resolveJuzIdentifier`). The app keeps this thin parsing layer and drives it with engine primitives + data (`juzPage`, `search`, `filterByCounts`, `surahFromEnd`/`juzFromEnd`, `ArabicText`, and `juz.json`'s `nameArabic`/`nameTransliteration`). The engine ships the primitives; the routing is UI.

## Don't-break checklist for the swap

- Tajweed output is pixel-identical: run the `tajweed` conformance vectors and a screenshot diff over a sample (esp. the tricky ones — madd 'iwad incl. the iqlaab form `رُوَيۡدَۢا`, muqattaat, hamzat-wasl, final-raa).
- Search returns the same hits in the same order (mushaf order) — the engine search is now byte-for-byte faithful to `QuranData.swift`.
- Counts (`juzStats`, word/letter/page) match.
